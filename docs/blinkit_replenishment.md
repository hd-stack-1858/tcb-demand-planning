# Blinkit Replenishment Model — Design Doc

Generated by /office-hours on 2026-05-20  
Branch: main | Status: APPROVED

---

## Problem Statement

The Cradle Box replenishes Blinkit inventory manually today: 6+ steps across multiple browser tabs — check availability% per SKU per city, look up warehouse inventory, count active dark stores per city, estimate 1/2/3 units per dark store on gut feel, verify total inventory available, then ship. At 6-7 cities (12 by June end), this process is slow, error-prone, and doesn't scale. The "gut feel" velocity step is the primary gap: each dark store has its own demand pattern, and the right data (ADS per dark store per day) already exists in Blinkit's performance evaluation export but isn't being used systematically.

---

## Current Manual Process (Status Quo)

6-step process run every replenishment cycle (~monthly per warehouse):

1. Check availability% per SKU per city in Blinkit portal
2. Check current warehouse inventory for that city
3. Count how many dark stores are active (enabled) for that SKU in that city
4. Estimate 1/2/3 units per dark store based on **gut feel** of sales velocity ← the gap
5. Verify total inventory available covers the combined send
6. Trigger shipment to Blinkit warehouse

Pain: multi-tab browser session, no data integration, gut-feel velocity, error-prone aggregation across cities. City-level velocity varies significantly; each dark store has its own demand pattern.

## Data Sources Available

| Source | What it contains | Where |
|--------|-----------------|-------|
| Performance data | ADS per dark store per day, availability%, complaint%, warehouse→darkstore mapping, open/closed status per SKU | `data/blinkit/manual/product_performance/` CSVs |
| Inventory snapshot | Units on hand per SKU per warehouse | `data/blinkit/manual/inventory/` CSVs |
| Sales data | Orders by city and SKU | Supabase prod DB (already loaded) |
| Selling prices | SP per SKU | Supabase prod DB (product master) |

**Performance data metrics (30-day evaluation cycles):**
- ADS (Average Daily Sales) = units sold ÷ days in stock (not total days — accurate velocity signal)
- Availability% — target >50%
- Complaint% — target <7%
- Dark store status: active, closed-by-Blinkit, or closed-for-SKU (low sales)

## Core Formula

```
For each SKU × Warehouse:

  active_dark_stores = dark stores in WH's area where status = active for that SKU
  
  target_stock = SUM(ADS_per_darkstore × 30) across active_dark_stores
  
  units_to_ship = max(0, target_stock - current_inventory_at_WH)

Total gate check:
  SP = selling price from Supabase product master (skus table, latest active price)
  invoice_value = SUM(units_to_ship × SP) across all SKUs in shipment
  if invoice_value < ₹2,00,000 → flag "below minimum — defer or batch"

Output behavior:
  - Rows where units_to_ship = 0 are SUPPRESSED by default (already well-stocked)
  - Pass --show-all flag to include zero-ship rows for review
```

Coverage target = 30 days (matches monthly shipping cadence; avoids overstock that drives up storage fees ₹20-23K/month).

## Premises & Constraints

| # | Premise | Notes |
|---|---------|-------|
| 1 | ADS from latest cycle = reliable velocity proxy | ADS counts only in-stock days, not OOS days — clean signal. **One-time validation required:** cross-check `ADS × in-stock_days` against actual Supabase order count for 2-3 SKUs before trusting in production. |
| 2 | Coverage = 30 days (1 month of expected sales) | Matches current monthly shipping frequency. As frequency increases (weekly at scale), reduce coverage window proportionally. |
| 3 | Minimum shipment gate = ₹2L invoice value | `Σ(units_to_ship × SP)`. Below this, logistics cost% is too high. Flag and defer, don't suppress. |
| 4 | New dark stores with <10-15 days data | Flag as `ESTIMATED` in output. Use best ADS of that SKU across active dark stores in the same city as the velocity estimate. If brand new city, use best ADS across any city. |
| 5 | Warehouse → dark store distribution lag | Blinkit controls internal allocation. Stock can sit at WH while dark stores are OOS for 5-7 days. Use model as **leading indicator** — ship 7-10 days before expected OOS, not as a real-time signal. |
| 6 | Not all SKUs in all cities | Performance data's active/closed status per SKU per dark store is the source of truth for inclusion. |

## Recommended Approach

**Performance data in Supabase + recommendation engine as Python module + pre-formatted RO draft output. Document automation (invoice, e-way bill) deferred.**

### Why this approach

- Performance data in DB (vs raw CSVs) enables cycle-over-cycle trend analysis and scales to 12 cities without extra manual steps per run
- Recommendation engine as a clean Python module is testable, reusable in Streamlit later
- Pre-formatted RO draft (Excel) reduces Blinkit portal upload to a 2-click import — 90% of the UX win without the fragility of full portal automation
- Third-party WMS (Unicommerce, EasyEcom) deferred: adds cost + complexity at a scale that doesn't justify it yet

### Data model

```sql
-- New tables to add to Supabase

blinkit_performance_cycles
  id, cycle_id, start_date, end_date,
  sku_id, dark_store_id, dark_store_name,
  warehouse_id, warehouse_name, city,
  ads_units,             -- avg daily sales in units (in-stock days only) ← used in formula
  -- ads_value (₹) not stored: redundant given SP in product master
  availability_pct, complaint_pct,
  status (active | closed_blinkit | closed_sku),
  data_quality_flag (ok | insufficient_data)

blinkit_inventory_snapshots
  id, snapshot_date, warehouse_id, city, sku_id, units_on_hand
```

### Python module structure

```
tcb/replenishment.py
  get_latest_performance_cycle(sku_id, warehouse_id) → DataFrame
    # "latest" = most recently ended cycle (end_date < today), tie-break on max end_date
  get_active_darkstores(cycle_id, sku_id, warehouse_id) → list[dark_store_id]
    # filters status='active' for that SKU; internally iterates over WH's dark stores
  calculate_coverage_need(sku_id, warehouse_id, days=30) → int
    # sums ADS across all active dark stores in WH, multiplies by days
  get_current_stock(warehouse_id, sku_id, snapshot_date=None) → int
    # snapshot_date=None → use most recent available snapshot date
  compute_replenishment_plan(snapshot_date=None, show_all=False) → DataFrame
    # show_all=False suppresses rows where units_to_ship=0
  check_minimum_gate(plan_df, min_value=200_000) → (bool, float)
  export_to_excel(plan_df, output_path, show_all=False) → None
```

## Open Questions

1. **ADS reconciliation (must-do before first production run):** Does `ADS × in-stock_days` from performance CSV match actual unit count in Supabase orders table for same period? Pick 2-3 SKUs in one city and verify.
2. **Multi-warehouse cities — PRE-BUILD BLOCKER:** Bangalore has 2 warehouses. The entire data model assumes `warehouse_id` exists in the performance CSV. Verify this field is present and populated before writing any code. If absent, the WH→dark store mapping must come from a separate source. Check one performance CSV manually first.
3. **New city launches (June) — DECIDED:** For dark stores with <10-15 days of ADS data, use **the best (highest) ADS for that SKU across all other active dark stores in the same city** as the estimate (flag these rows in output as `ESTIMATED`). If no city-level ADS exists (brand new city), use the best ADS for that SKU across any city. Revisit after first full cycle completes.
4. **Performance cycle boundaries — CONFIRMED:** Cycles are 30-day windows, start/end dates tied to SKU launch date (not calendar months). A given SKU's cycles are sequential with no overlap. Cycles across different SKUs have overlapping dates. "Latest cycle" for a SKU = the most recently *ended* 30-day window for that specific SKU (max `end_date` where `end_date < today` and `sku_id = target`). Never use an open/active cycle.

## Success Criteria

- Replenishment calculation time: from ~2 hours of manual tab-switching to <15 minutes (load snapshot + run script + review output)
- Model output validated against last 2 actual shipments (back-test before first live use)
- ₹2L gate correctly flags under-threshold shipments
- Performance data for all past cycles loaded into Supabase before June inventory arrives
- First automated replenishment run completed for June restock

## The Assignment (do this before writing code)

Run the replenishment formula manually for **one city** (Bangalore or Hyderabad — whichever has the most consistent performance data). Use the ADS from the most recent performance cycle CSV + latest inventory snapshot. Compare the output to your last actual manual replenishment decision for that city.

- If they're within 20%: model is calibrated, proceed to build
- If they diverge significantly: you've found the premise that needs fixing before building the pipeline

This takes 30 minutes in a spreadsheet. Do it before writing a single line of code.
